# 改善記録

## 2026-05-31

### 方針決定
- 学習記録の正の保存先はSupabase Postgresにする。
- GitHubはコード、仕様、DBマイグレーション、変更履歴の管理に使い、ユーザーの学習記録そのものは保存しない。
- ブラウザのlocalStorageは、オフライン時や同期失敗時の一時保存として残す。
- ログインは、管理者が事前に登録したメールアドレスだけを許可する方針にする。
- 初回利用時のQRコードは「アプリを開くための導線」とし、毎回のログイン手段にはしない。
- 2回目以降は、Supabase Authのセッションが残っていればアプリアイコンやブックマークからそのまま使える形を目指す。

### 実装した改善
- Supabase保存用のDBマイグレーション案を追加した。
- ユーザー本人は自分の学習記録だけ、管理者は全ユーザーの記録を見られるRLS方針を追加した。
- 学習進捗、日別学習記録、プロフィールをSupabaseに保存する設計に更新した。
- アプリ側にSupabase同期処理を追加した。
  - `user_app_state` に現在章、章ごとの進捗、単語カード、アウトプット進捗、リマインド、質問メモを保存する。
  - `daily_learning_records` に日別の学習時間、解答数、正答数、完了章数を保存する。
  - 同期できない場合はlocalStorageに一時保存する。
- ログイン画面をメールアドレス + 6桁コード方式に変更した。
- 管理画面はSupabase上のプロフィール・学習記録を読める場合、サーバー側の全ユーザー記録を優先表示するようにした。
- スマホ利用を前提に、PWA用のmanifest、service worker、ホーム画面アイコンを追加した。
- Supabase Edge Functionでmanifest、service worker、アイコンを配信できるようにした。

### 未完了・次に確認すること
- Supabase本番REST APIで確認したところ、`profiles`、`user_app_state`、`daily_learning_records` はまだ存在しないため、Supabase DashboardでマイグレーションSQLを適用する必要がある。
- 管理者ユーザーの `profiles.role` を `admin` に更新する必要がある。
- Supabase Authのメールテンプレートを、Magic Linkか6桁OTPのどちらで運用するか決める必要がある。
- 管理画面からAuthユーザーを作成するには、service role keyを使うEdge Functionが必要。
- ローカルにはSupabase Access Token、DB接続文字列、DBパスワードが見つからなかったため、DB反映作業には追加情報が必要。

## 2026-06-20 本番反映準備

### 確認したこと
- Supabase CLIは `npx supabase` で利用できる。
- ローカルにはSupabase Access Tokenが未設定。
- 本番DBへマイグレーションを反映するためのDBパスワードまたはDB接続URLも未設定。

### 追加したもの
- `scripts/deploy-supabase-production.ps1` を追加した。
- 秘密情報をリポジトリに保存せず、環境変数から本番反映できるようにした。

### 必要な情報
- `SUPABASE_ACCESS_TOKEN`
- `SUPABASE_DB_PASSWORD` または `SUPABASE_DB_URL`
- 必要に応じて `SUPABASE_SERVICE_ROLE_KEY`
